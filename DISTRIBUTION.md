# Distributing Quillery

Quillery ships two ways: as a **browser extension** (Chrome Web Store / Edge
Add-ons) and as a **Windows desktop app** (Electron installer). This guide
covers both, and flags the steps only you can do (they need accounts, payment,
or a signing certificate).

---

## A. Browser extension — Chrome Web Store

### 1. Build the upload package
```bash
npm run build:extension     # produces dist-extension/
npm run zip:extension       # produces release/quillery-extension-v<version>.zip
```
The `.zip` is what you upload. (You can also zip `dist-extension/` manually.)

### 2. One-time setup (you must do this)
- Create a **Chrome Web Store developer account**: https://chromewebstore.google.com/devconsole — a **one-time $5 USD** registration fee.
- Host the **privacy policy** at a public URL. Easiest options:
  - Push this repo to GitHub and enable GitHub Pages, then link `PRIVACY.md`, or
  - Paste `PRIVACY.md` into a free page (e.g. a GitHub Gist, Notion public page,
    or your own site). You'll paste this URL into the listing.

### 3. Create the listing
- Dashboard → **New item** → upload the `.zip`.
- Fill **Store listing** from [`docs/STORE-LISTING.md`](docs/STORE-LISTING.md)
  (name, summary, description, category, single-purpose, screenshots).
- Fill **Privacy practices**: declare **no data collected**, add the privacy
  policy URL, and justify the single `sidePanel` permission.
- Upload the **128×128 icon** (`public/icons/icon128.png`) and **1–5
  screenshots** at 1280×800.

### 4. Submit
- Submit for review. First reviews typically take a few days. Because Quillery
  requests only `sidePanel` and collects no data, it should review cleanly.

### Edge Add-ons (bonus, same package)
The exact same `.zip` works for the **Microsoft Edge Add-ons** store
(https://partner.microsoft.com/dashboard/microsoftedge) — registration there is
**free**. Worth doing since the side panel works in Edge too.

## B. Desktop app — Windows installer (Electron)

### 0. One-time prerequisite (Windows symlink privilege)
electron-builder extracts a signing-tools cache that contains symlinks. Creating
symlinks on Windows needs a privilege normal accounts lack, so the first
`electron:build` can fail with:
`Cannot create symbolic link : A required privilege is not held by the client`.
Fix it once, either way:
- **Enable Developer Mode:** Settings → Privacy & security → For developers →
  **Developer Mode = On** (grants the symlink privilege to your user), **or**
- Run the build from a terminal opened **as Administrator**.

The Vite/Electron app bundle itself builds fine without this — only the
electron-builder packaging step needs it.

### 1. Build the installer
```bash
npm run electron:build      # icons + vite build + electron-builder
```
Output lands in `release/`:
- `Quillery Setup <version>.exe` — the NSIS installer you distribute.
- An unpacked app under `release/win-unpacked/` (for quick local testing).

The installer icon and in-app icon come from `public/icons/icon.ico` /
`dist/icons/icon256.png`, both generated by `npm run icons`.

### 2. Code signing (recommended — you must obtain a certificate)
Unsigned installers run, but Windows SmartScreen shows a "unknown publisher"
warning until the app builds reputation. To sign:
- Buy an **OV or EV code-signing certificate** (e.g. from a CA like DigiCert,
  Sectigo). EV certificates clear SmartScreen immediately; OV builds reputation
  over time.
- Then either set the env vars electron-builder reads
  (`CSC_LINK` = path to your `.pfx`, `CSC_KEY_PASSWORD` = its password) before
  running `electron:build`, or add a `win.certificateFile` /
  `certificatePassword` entry to the `build` block in `package.json`.

### 3. Distribution channels
- Host `Quillery Setup <version>.exe` on a download page / GitHub Releases, or
- Submit to the **Microsoft Store** (needs a Partner Center account, one-time
  fee) — this also removes the SmartScreen warning.

### macOS / Linux (later)
The same React build runs on macOS/Linux; add `mac`/`linux` targets to the
electron-builder `build` block and build on (or via CI on) those platforms.
macOS distribution requires an Apple Developer account ($99/yr) for notarization.

## Versioning checklist (each release)
Bump the version in **both** places so the targets stay in sync:
- `package.json` → `version`
- `public/manifest.json` → `version`

Then rebuild the target(s) and re-upload.

## Quick reference — what only you can do
| Step | Needs |
| --- | --- |
| Publish to Chrome Web Store | $5 one-time dev account |
| Publish to Edge Add-ons | Free dev account |
| Host the privacy policy | A public URL (GitHub Pages / Gist / site) |
| Sign the Windows installer | A code-signing certificate |
| Microsoft Store / Apple notarization | Store dev accounts |
